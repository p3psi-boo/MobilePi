# 项目需求文档 (PRD)：移动端多机 Coding Agent 调度编排系统

## 1. 项目概述与核心定位

本系统旨在为开发者提供一个“离屏自治”**的移动端项目推进器。系统的核心定位是**“指挥官看板”而非“移动端 IDE”。
用户在脱离开发环境（无电脑）时，可通过该系统在 Android 手机或 iPad 上向分布在多台物理机（Mac/Linux）上的 Coding Agent（如 CodeX、PI Coding Agent）下达自然语言指令、接收富媒体运行结果（截图/URL）、并进行关键节点决策。

* **核心准则：** 彻底屏蔽代码编辑器（零代码编辑功能），用户仅通过“语义指令”与“上下文回滚”对 Agent 进行纠偏。
* **技术栈约束：** 全栈使用 Dart 语言开发（Flutter + Dart CLI Server + Dart AOT Node）。

---

## 2. 系统整体架构 (三端分离)

系统采用 **“拉取式 (Pull-Based)”** 和 **“极简中枢 (Thin Hub)”** 的三层协同架构：

### 2.1 Client 端 (控制台客户端)

* **运行平台：** Android (手机) 与 iPadOS (平板)。
* **核心职责：** 连接 Hub 发起拉取请求同步已注册 Daemon 状态、下达决策指令、渲染看板与对话流、呈现运行截图/预览。
* **边界约束：** Client 不负责添加 Daemon，也不暴露 Daemon 连接方式编辑能力；Daemon 接入由 Daemon 端主动连接 Hub 完成。

### 2.2 Hub 端 (中枢转发服务器)

* **运行平台：** 云端服务器。
* **核心职责：** 作为 WebSocket 中枢，接受 `Client -> Hub` 与 `Daemon -> Hub` 两类连接，维护内存级 Daemon 路由表，并作为 Daemon 节点二进制更新包的文件宿主 (HTTP File Server)。
* **数据约束：** **不持久化存储任何业务数据与对话日志**，仅维持内存级的 Daemon 路由表与最新状态快照。

### 2.3 Daemon 端 (物理机执行节点)

* **运行平台：** macOS 与 Linux（支持 ARM64 与 x86_64 架构）。
* **核心职责：** 作为 Agent 的守护进程 (Wrapper)，主动连接并注册到 Hub。负责调用本地 Agent 命令行、扫描和读取本地会话日志、持久化存储历史状态、截取运行结果，并执行自动更新。

---

## 3. 核心功能需求

### 3.1 Client 端 (交互与决策层)

* **基于状态的任务看板：**
* 摒弃传统按时间排序的 IM 列表，采用**按任务状态分类**的列表机制（如：`Running` 进行中、`Waiting` 待决策、`Idle` 闲置）。
* 列表卡片需展示提取自 Node 端的摘要级数据（如：任务运行进度、剩余时间估算）。


* **快捷指令盘 (Action Palette)：**
* 禁止用户高频输入斜杠命令（`/`）。
* 提供支持模糊搜索的动态快捷按钮组（如：`Deploy`, `Test`, `Undo`）。按钮需具备上下文感知能力，根据当前 Agent 状态自动更新选项（如：报错时自动提示 `[查看日志]`、`[换思路重试]`）。


* **“看门狗”状态监控与视觉反馈：**
* 提供明确的视觉指示灯（呼吸灯/标签）：正常运转为绿色/蓝色，等待用户确认为黄色，死循环或异常报错为红色。
* **一键紧急停止 (Panic Button)：** 提供全局/单项目的一键中断功能，强制 Node 端结束当前 Agent 进程。


* **富媒体产出物预览：**
* 不支持查看具体代码 Diff，仅显示增删行数作为参考。
* 支持拉取 Node 端传回的运行结果截图、日志摘要或 Web 预览 URL，并在 App 内直接渲染。



### 3.2 Node 端 (执行与存储层)

* **环境与状态持久化：**
* 在本地维护微型账本（如 JSON/SQLite），持久化存储对话与执行日志，作为 Client 掉线重连后的唯一数据源 (Source of Truth)。


* **Agent 隔离与对接：**
* 支持 CodeX 与 PI Coding Agent (Pi)。
* 通过子进程调用 (Process.start) 执行命令行工具。


* **会话管理 (Session Management)：**
* **会话扫描：** Node 需具备读取对应目录（如 CodeX 的 `~/.codex/sessions/` 与 Pi 的 `~/.pi/agent/sessions/`）权限，扫描现有 `.jsonl` 或会话树文件。
* **会话恢复：** 支持根据 Client 传入的 `Session_ID`，拼接原生恢复命令（如 `codex resume <ID>` 或 `pi --session <ID>`）唤醒指定会话。


* **侧车式自动更新 (Sidecar Auto-Update)：**
* 提供独立的 Launcher 进程，通过对比版本号，主动从 Hub 的 HTTP 接口下载对应架构的新版二进制文件。
* 自动执行文件替换、权限赋予 (chmod +x) 及平滑重启，支持失败回滚。



### 3.3 Hub 端 (路由与分发层)

* **安全与通信：**
* 全程采用 TLS 加密。
* 使用 WebSocket 建立两条独立长连接：`Client -> Hub` 与 `Daemon -> Hub`。Client 与 Daemon 不直接建连。


* **路由寻址：**
* 基于设备的全局唯一 ID 进行消息投递，不解析具体的 JSON Payload（版本更新协商指令除外）。
* Daemon 连接 Hub 后主动发送状态快照完成注册；Client 的同步请求只能返回 Hub 当前已注册的 Daemon。
* Client 指令必须带目标 Daemon ID，由 Hub 根据内存路由表转发；Client 不能新增 Daemon，也不能修改 Daemon 的 Hub 连接配置。


* **静态文件宿主：**
* 编译期间，自动将不同平台的 Node 可执行文件挂载至 Hub 的 Web 目录下，供 Node 自动拉取更新。

## 4. 多端设备与布局需求

* **Android 端 (手机形态)：**
* 优化单手操作体验。
* 采用底部 Tab 导航，支持下拉强制刷新获取全量进度，前台驻留时定期（差量）静默拉取。


* **iPad 端 (大屏指挥官形态)：**
* 强制采用**双栏/分屏布局 (Master-Detail)**。
* 左侧区域：全局 Node 节点机器列表、按状态分类的频道树状图。
* 右侧区域：Agent 沉浸式对话流、大图预览及内网穿透 Webview 展示。


* **跨设备视觉隔离：**
* 针对不同的物理机 Node，分配强制性的视觉差异标识（如图标/环境色调）。
* 针对不同的 Agent（CodeX vs Pi），分配不同的对话气泡风格或头像，杜绝混淆。



---

## 5. 极端场景与鲁棒性约束

1. **断网与后台存活：**
* Client 在网络恢复或从后台切回前台时，需通过 Daemon 端的 `last_msg_id` 机制，经 Hub 快速拉取缺失状态（差量同步），确保进度连续性。


2. **错误处理的非人工干预：**
* 当出现代码冲突或复杂逻辑报错时，系统需提供“意图语义修正”方案（如：告知 Agent “放弃新功能，回退上一版本”），严禁引导用户进入代码层级进行人工 Debug。


3. **大流量防卡顿策略：**
* 下拉同步数据时，必须将“富媒体流”（截图/大文件）与“文本流”解耦。默认仅拉取图片 URL 或摘要，用户主动点击后才请求二进制大文件，保障弱网环境下的任务推进效率。
